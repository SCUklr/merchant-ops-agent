# 电商智能客服 AI Agent 项目技术规格文档

## 1. 文档目标

本规格文档用于指导项目按 Spec-Driven 方式落地，并作为面试可追问的统一技术依据。文档目标：

- 明确业务边界、系统架构、核心链路和风险控制方案。
- 固化 8 步渐进式开发计划，每步均可独立验收。
- 为简历和面试提供可自证的技术与指标口径。

---

## 2. 业务问题与系统目标

### 2.1 业务问题

电商商家日常运营常见问题：

- 商品规则问答：发货时效、退换货、平台规则解释。
- 活动 SOP 查询：大促活动配置、履约流程、质检流程。
- 订单异常定位：超时未发货、物流停滞、退款冲突。
- 评价归因与建议：差评原因识别、补救动作推荐。

### 2.2 系统目标

将问题处理流程标准化为：

`用户问题 -> 意图识别 -> RAG 检索 -> Tool 调用 -> 规则校验 -> 可执行响应`

目标收益：

- 降低人工检索和跨系统跳转成本。
- 提升回答一致性、可解释性和可追溯性。
- 支持可观测、可回测、可持续迭代。

---

## 3. 非功能性要求

- **可用性**：核心 API 可稳定返回，失败有降级输出。
- **可观测性**：每次对话具备 trace_id、阶段耗时和结果状态。
- **安全性**：对高风险结论启用规则拦截，避免直接执行危险动作。
- **可测试性**：至少具备 smoke + case 回放 + 回归对比能力。
- **可维护性**：分层清晰，模块输入输出契约明确，便于替换模型与工具。

---

## 4. 高层架构

### 4.1 架构分层

- **接入层**：FastAPI（REST + SSE）。
- **编排层**：Agent Orchestrator（意图识别、任务拆解、阶段执行）。
- **能力层**：
  - RAG 子系统（切分、索引、召回、重排、引用）。
  - Tool 子系统（订单查询、工单创建、通知推送）。
  - Guardrails（规则校验、回退策略、风险提示）。
- **数据层**：Redis（缓存/会话）、MySQL（业务数据）、Elasticsearch（全文检索）。
- **评测层**：离线 case 集、端到端任务回放、指标统计。

### 4.2 核心交互时序（简化）

1. 接口接收问题并生成 `trace_id`。
2. 编排器分类：问答类 / 异常定位类 / 执行类。
3. 问答类进入 RAG；执行类触发工具调用计划。
4. Guardrails 扫描输出风险等级和缺失条件。
5. SSE 流式返回阶段信息 + 最终回答 + 依据。

## 5. 领域模型与接口契约

### 5.1 核心请求对象

- `ChatRequest`
  - `user_id: str`
  - `session_id: str`
  - `message: str`
  - `context: dict[str, str] | None`

### 5.2 核心响应对象

- `ChatFinalResponse`
  - `trace_id: str`
  - `intent: str`
  - `answer: str`
  - `evidence: list[str]`
  - `actions: list[dict]`
  - `risk_level: str`

### 5.3 流式事件对象

- `phase`: `intent` / `retrieve` / `tool` / `guard` / `final`
- `content`: 当前阶段输出
- `meta`: 阶段耗时、命中数、工具名等

## 6. 核心链路规格

### 6.1 商品规则问答（RAG）

- 输入：自然语言规则问题。
- 处理：
  - Query 重写；
  - 召回（向量 + 关键词）；
  - 重排（语义相关性）；
  - 输出答案并附证据片段。
- 验收：返回答案必须包含至少 1 条证据来源。

### 6.2 订单异常定位（Tool + Rule Check）

- 输入：订单号 + 异常描述。
- 处理：
  - 调用订单查询工具；
  - 归因模板推断（物流、库存、支付、客服）；
  - 规则校验可执行性（是否满足操作前置条件）。
- 验收：给出异常原因、建议动作、执行限制。

### 6.3 工单创建与通知（Function Calling）

- 输入：执行意图（如“帮我建工单并通知仓库”）。
- 处理：
  - 生成工具调用计划；
  - 按顺序执行 `create_ticket`、`push_notification`；
  - 汇总执行结果和追踪号。
- 验收：工具返回结构化结果，可追踪。

## 7. Prompt 与 Context Engineering 规范

### 7.1 系统 Prompt 基线

- 明确身份：电商运营助手，不编造规则。
- 明确流程：先检索/查询，再给结论。
- 明确边界：缺失关键参数时先澄清，不擅自执行。
- 明确输出：给结论 + 证据 + 后续动作。

### 7.2 上下文组织策略

- 固定区块：`system`、`conversation_summary`、`latest_user_goal`、`retrieved_evidence`、`tool_result`。
- 长对话策略：历史消息摘要化，仅保留当前任务强相关上下文。
- 防污染：检索片段与用户输入隔离，避免拼接注入。

### 7.3 风险应对策略

- 幻觉：无证据时降级为“待确认建议”。
- Prompt 注入：过滤“忽略系统指令”等特征。
- 上下文污染：对工具输出做 schema 验证，不信任原始文本。

## 9. 8 步开发计划（总览）

1. 基础工程骨架迁移并跑通最小 API。
2. 接入统一对话编排器与流式事件协议。
3. 完成商品规则问答 RAG 最小闭环。
4. 完成订单异常定位 Tool + Rule Check 链路。
5. 完成工单创建/通知 Function Calling 链路。
6. 增加会话记忆、缓存与故障回退策略。
7. 建立评测回放框架（10-20 case）。
8. 产出可自证指标与面试答辩文档。

---

## 10. 指标口径（面试可自证）

- **检索命中率**：标注答案是否在召回 TopK 中。
- **端到端成功率**：无需人工介入完成目标任务的比例。
- **处理时延**：从请求到可执行建议输出的总耗时。
- **工具成功率**：工具调用成功次数 / 总调用次数。

所有指标必须记录样本范围、统计周期和基线定义。
